## 属性 (Properties)

| 属性名        | 类型       | 必需 | 描述                               |
| ------------- | ---------- | ---- | ---------------------------------- |
| `label`       | `string`   | ✅   | 更新标签，显示在左侧并创建锚点链接 |
| `description` | `string`   | ❌   | 更新描述，显示在标签下方           |
| `tags`        | `string[]` | ❌   | 标签列表，显示为右侧面板的过滤器   |

## 使用建议

### 变更日志结构

建议按照以下结构组织变更日志：

```mdx 推荐的变更日志结构
<Update label="YYYY-MM-DD" description="版本号" tags={["类型"]}>
  ## 简短描述

### 新功能 (New Features)

- 功能描述

### 功能改进 (Improvements)

- 改进描述

### 问题修复 (Bug Fixes)

- 修复描述

### 破坏性变更 (Breaking Changes)

- 变更描述和迁移指南

### 弃用通知 (Deprecations)

- 弃用功能和替代方案

</Update>
```

### 标签分类建议

常用的标签分类：

- **`Feature`** - 新功能
- **`Improvement`** - 功能改进
- **`Bug Fix`** - 问题修复
- **`Performance`** - 性能优化
- **`Security`** - 安全修复
- **`UI/UX`** - 界面和体验改进
- **`API`** - API 相关变更
- **`Documentation`** - 文档更新
- **`Major Release`** - 重大版本发布
- **`Hotfix`** - 紧急修复

### 内容编写指南

1. **清晰的标题** - 使用描述性的标题总结主要变更
2. **结构化内容** - 使用标题和列表组织内容
3. **用户视角** - 从用户角度描述变更的影响和价值
4. **包含截图** - 对于 UI 变更，提供前后对比截图
5. **迁移指南** - 对于破坏性变更，提供详细的迁移步骤

更新组件是维护产品发布记录和与用户沟通的重要工具。通过一致的格式和清晰的内容，可以帮助用户了解产品的演进历程和最新功能。
